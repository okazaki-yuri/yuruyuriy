# 設計書 : 多言語対応（i18n）基盤

サイト全体の多言語対応の実装アーキテクチャを記載する。方針・経緯・フェーズ計画は [i18n-plan.md](./i18n-plan.md) を参照。

- 現在の公開言語: **日本語（ja）・英語（en）**。日本語はルート直下、英語は `/en/` 配下（Phase 2 で追加済み）。
- URL 設計: **A案**。日本語はルート直下（現行 URL 維持）、英語は `/en/` 接頭辞（例: `/en/tools/`）。

## 1. ディレクトリ構成

```
apps/web/app/
├── i18n/                  … i18n 基盤
│   ├── types.ts           … Locale 型・Dictionary 型（全文言のスキーマ）
│   ├── ja.ts              … 日本語辞書（satisfies Dictionary でキー過不足を検出）
│   ├── en.ts              … 英語辞書（同上）
│   └── index.ts           … getDictionary(locale) / localizePath(locale, path)
├── content/               … ページの実体（言語間で共有。locale を受けて辞書から文言を引く）
│   ├── root.tsx           … ルートレイアウトの実体（RootDocument / buildRootMetadata / viewport）
│   ├── home.tsx           … トップ（HomeContent / buildHomeMetadata）
│   ├── tools.tsx          … ツール一覧
│   ├── contact.tsx        … お問い合わせ（contact.css も隣置）
│   ├── legal.css          … 法務ページ共通CSS（(ja)/(en) 両方の legal ページから import）
│   ├── wordroulette/      … ことばルーレットちゃん（page-content.tsx / WordRoulette.tsx / RouletteWheel.tsx / roulette.css）
│   └── webdice/           … WEBサイコロちゃん（page-content.tsx / WebDice.tsx / dice.css）
├── (ja)/                  … 日本語ルート（Route Group。URL に現れない）
│   ├── layout.tsx         … 日本語ルートレイアウト（<html lang="ja">。実体は content/root.tsx）
│   ├── page.tsx           … 各ページは content/ を locale='ja' で呼ぶ薄いラッパー
│   ├── tools/… contact/…
│   └── legal/             … 法務ページ（本文はロケール別ファイルとして保持。辞書化しない）
├── (en)/                  … 英語ルート（Route Group）
│   ├── layout.tsx         … 英語ルートレイアウト（<html lang="en">。実体は content/root.tsx）
│   └── en/                … /en/ 配下の各ページ（content を locale='en' で呼ぶ薄いラッパー）
│       └── legal/         … 法務ページ英語版（独立した英訳。「日本語版が正文・優先」の Language 条項付き）
├── global-not-found.tsx   … グローバル404（後述）
├── components/            … 共通コンポーネント（Header / Footer / ShareButtons は locale を受け取る）
├── sitemap.ts robots.ts manifest.ts … ルート直下必須のメタデータルート（言語共通で1つ）
└── site.ts og.ts styles/
```

- **さらに言語を追加する際**は、`Locale` 型と core の `SupportedLocale` に列挙を足し、辞書（`i18n/<locale>.ts`）・Route Group（`app/(<locale>)/…`）・法務ページ訳を同じパターンで追加する。
- **Web App Manifest（PWA）は主言語（ja）の文言で1枚のみ**（静的エクスポートでは `/manifest.webmanifest` を言語別に出し分けられない）。`/en/` からインストールした場合もアプリ名は日本語になる（既知の制約）。文言は `manifest.ts` にハードコードせず、ja 辞書（`getDictionary('ja')`）から取得する。

## 2. 辞書（Dictionary）

- 全 UI 文言・メタデータ文言・JSON-LD 文言・FAQ・confirm/alert・placeholder・aria-label を `i18n/ja.ts` に集約。
- **言語ごとの機能フラグも辞書で持つ**: `header.showBlog`（ブログは日本語のみのため en は false）/ `share.showLine`（LINE は日本・アジア圏中心のため en は false。アジア圏言語の追加時は true にする）。
  - `showBlog` が制御するのは**ヘッダーのナビリンクのみ**。トップページのブログボックスは**全言語で表示する（意図的な仕様）**。ブログ自体は日本語のみだが、トップではサイト紹介の一部として案内し、en では辞書の文言内の「(In Japanese)」注記で言語を明示する。
- `Dictionary` 型（`i18n/types.ts`）が文言スキーマ。各ロケール辞書は `satisfies Dictionary` で定義するため、**キーの過不足はコンパイルエラー**になる。
- 数値等の埋め込みが必要な文言は**関数**として持つ（例: `durationSeconds: (n) => \`${n}秒\``）。英語の複数形にも関数内で対応できる。
- 改行（`<br>`）を含む段落は `string[]`（行の配列）で持ち、`components/MultilineText.tsx` で描画する。
- 文中リンクは `LinkedText`（`{ pre, linkText, post }`）で持つ。
- **CSS の疑似要素（`::before` 等）で表示する文言も CSS にハードコードしない**。コンポーネントが辞書の値を `data-*` 属性で要素に付与し、CSS は `content: attr(data-*)` で参照する（例: ルーレット結果ボックスの `data-placeholder` / `data-spinning`）。
- **クライアントコンポーネントは `locale` を props で受け取り、`getDictionary(locale)` を自身で呼ぶ**（辞書に関数が含まれるため、Server→Client に辞書オブジェクトを渡さない）。

## 3. URL・リンク

- 内部リンクは必ず `localizePath(locale, path)` を通す（ja は素通し、en は `/en` 接頭辞）。`data/tools.json` の `href` はロケール接頭辞なしで持つ。
- canonical / OGP url / JSON-LD の URL も `localizePath` で組み立てる（`site.ts` の `buildOpenGraph` は `locale` 必須）。`buildOpenGraph` は `og:locale`（`ja_JP` / `en_US`。`site.ts` の `OG_LOCALES`）と `og:locale:alternate` も出力する。ロケール追加時は `OG_LOCALES` にも列挙する。
- **言語スイッチャー**: `components/LanguageSwitcher.tsx`。`usePathname()` で現在パスからロケール接頭辞を除いた共通パスを求め、全言語（`LOCALES`）の対応ページへリンクする。言語名は `i18n/index.ts` の `LOCALE_NAMES`（常にその言語自身の表記）。PC ヘッダーは 🌐 ドロップダウン（`variant='dropdown'`）、モバイルメニューは通常項目として全言語を表示（`variant='menu'`、現在言語は `.lang-current` で強調）。スタイルは `styles/header-footer.css`。
- **OGP 画像**: 画像本体（ロゴ + ドメイン表記）は言語共通で、`og.ts` の `OG_IMAGE`（ja）/ `OG_IMAGE_EN`（alt のみ英語）を `buildOpenGraph` がロケールで出し分ける。

## 4. SEO（hreflang / sitemap）

- **hreflang**: 全ページの `alternates` は `site.ts` の `buildAlternates(locale, path)` で組み立てる。canonical は自ページ自身、hreflang は `ja` / `en` / `x-default`（日本語＝主言語）の3本を出力する。ページ追加時も必ずこのヘルパーを使う。
- **sitemap**（`app/sitemap.ts`）: 各パスにつき日本語・英語の2エントリを登録し、それぞれに hreflang の相互参照（`alternates.languages` → `xhtml:link`）を付ける。
- **lastmod は言語別管理**: `lastmod: { ja, en }` の形で持つ。片方の言語だけ更新した場合はその言語の lastmod のみ書き換える。全ページに影響する変更（ヘッダー・共通CSS等）は全エントリ・全言語を更新する。
- **JSON-LD**: `inLanguage` は locale から出し分け（content コンポーネント内）。FAQ は表示文言と FAQPage の一致を両言語で維持する。

## 5. packages/core の多言語対応

- `tools.json` の `name` / `description` は `{ "ja": "...", "en": "..." }` 形式（`LocalizedString`）。en 未翻訳の間は省略可で **ja へフォールバック**。
- `toolsRepository.getTools(locale)` / `getTool(slug, locale)` がロケール解決済みのフラットな `Tool` を返す（既定は `'ja'`）。
- `validateDiceConfig` は**エラーコード**（`DiceValidationError`）を返す。表示文言は web 側辞書（`dice.widget.errors`）で解決する。core は UI 文言を持たない。
- `sortWords(words, order, locale)` … 比較ロケールを引数で指定（既定 `'ja'`）。

## 6. グローバル 404（global-not-found）

- 複数ルートレイアウト構成ではルート直下に `layout.tsx` が無いため、通常の `not-found.tsx` はレイアウト（`<html lang>`・ヘッダー・CSS）の外で描画されてしまう。
- そのため **experimental の `global-not-found.tsx`**（`next.config.js` の `experimental.globalNotFound: true`）を使い、`RootDocument` で完全なドキュメントを自前描画する。
- 静的エクスポートでは `out/404.html` は1枚のみのため、**日英併記**にしている（日本語の案内 + `lang="en"` 付きの英語の案内。ページ枠は ja）。404 は noindex のため sitemap・hreflang の対象外。noindex は **Next が not-found ページへ自動付与するものに任せ、`metadata.robots` は指定しない**（指定すると `<meta name="robots">` が重複出力される）。
- レイアウトのメタデータ・title テンプレート・viewport を**継承しない**ため、`buildRootMetadata('ja')` を土台に自前で組み立てている。
- ⚠️ experimental 機能のため、**Next.js アップグレード時は 404 ページ（`out/404.html` のレイアウト適用と、robots メタが `noindex` 1つだけ出力されていること）を必ず確認**する。

## 7. 文言変更時の運用

- 表示文言の変更は **`i18n/ja.ts` と `en.ts` の両方を編集**する。ページ JSX（content/）には文言を直接書かない。
- 例外: 法務ページ（`app/(ja)/legal/` と `app/(en)/en/legal/`）は本文をロケール別ファイルとして持つ（辞書化しない）。**日本語版を改定したら英語版も追随させる**（日本語版が正文）。
- 文言変更時は該当ページ・**該当言語**の `sitemap.ts` の `lastmod` 更新（§4）・CHANGELOG 追記のルール（CLAUDE.md）は従来どおり。
